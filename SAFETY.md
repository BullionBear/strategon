# 策略發佈平台 — 安全與正確性邊界

> 版本 v0.1 · 搭配 ARCHITECTURE.md / PROTOCOL.md / RECONCILER.md 閱讀
>
> 本文件處理整個系統唯一一類「錯了會直接虧錢」的問題:**同一策略在兩台機器同時下單(雙開)**,以及與之交纏的時鐘假設、遷移互鎖、權限爆炸半徑。前三份文件描述系統如何運作;本文件描述它在什麼假設下才**安全**,以及假設破裂時的殘餘風險。

---

## 0. 一句話立場

**平台無法「消除」雙開,只能把它收斂到有界且可接受的殘餘風險。** 任何聲稱絕對防雙開的說法都是錯的,因為真正的 fencing 要求下游(交易所訂單閘道)驗證 fencing token 並拒絕過期者,而交易所不是我們的,拿不到這個配合。本文件的目的是把「有界」的界講清楚,並列出把界收窄的每一個手段。

---

## 1. 威脅模型:雙開如何發生

策略 S 原本在機器 M1 上運行並持有 lease。目標是:當 M1 出問題、S 被遷到 M2 時,**絕不存在 M1 與 M2 同時對交易所送單的時刻**。

雙開的根源是 M1「其實還能下單」但系統以為它死了。三種路徑:

| 路徑 | 機制 | lease + 自殺是否擋得住 |
|---|---|---|
| **A. 網路分區** | M1 連不上控制面,但 M1↔交易所仍通 | ✅ 擋得住(§2),前提是時鐘假設成立 |
| **B. 進程凍結** | VM suspend / 超長 GC / kernel stall;M1 既不續約也不感知時間 | ⚠️ 部分,有殘餘風險(§3) |
| **C. 腦裂誤判** | 控制面誤判 M1 死亡並過早指派 M2 | ✅ 擋得住,靠 §4 的遷移互鎖 |

lease 機制對 A 和 C 是健全的;C 只要遷移互鎖正確。**B 是唯一無法純靠本平台完全消除的**,因為它攻擊的正是 lease 賴以成立的前提——「時間在流動且 M1 感知得到」。

---

## 2. 基準機制:lease + 自殺(對路徑 A)

回顧 PROTOCOL §7.1 與 RECONCILER §8 的機制,並補上它成立的**時序條件**:

```
M1 持有 lease,TTL = T,須在到期前續約
M1 分區 → 續約失敗 → M1 在本地時鐘的 (取得時刻 + T - margin_agent) 自殺策略 S
控制面在 (最後一次成功續約 + T + margin_cp) 之後,才允許把 S 指派給 M2
```

**安全的充分條件**:

```
margin_agent + margin_cp  >  兩端最大時鐘偏差 + 續約 RTT 抖動
```

即 M1 必須在控制面認為 lease 過期**之前**就已自殺完成(含撤單/平倉 drain)。只要這個不等式成立,M1 停單的時刻嚴格早於 M2 開單的時刻,無重疊。

**這裡有一個必須明說、之前隱含的假設**:上式依賴「M1 的本地時鐘在流動且與控制面偏差有界」。路徑 A(單純網路分區)下這成立——CPU 還在跑,monotonic clock 照走。路徑 B 下這**不成立**,見下節。

---

## 3. 路徑 B:進程凍結下的殘餘風險(核心難點)

### 3.1 為什麼 lease 會破功

VM suspend 或等價的凍結期間,M1 的 monotonic clock **一起凍住**。M1 解凍時:

1. 它以為只過了一瞬,相信自己仍持有 lease(因為本地時鐘顯示還沒到期)。
2. 此時控制面那邊的 wall-clock 時間可能已經過了 T,lease 早已過期,M2 可能已經啟動 S 並在下單。
3. M1 在**續約迴圈下一次醒來察覺 lease 已死之前**,若策略主迴圈搶先送出一筆單——那一筆就是雙開窗口。

根因:lease 的安全性建立在「M1 能感知時間流逝從而主動自殺」,而凍結精準地破壞了這個感知。背景續約迴圈是週期性的,凍結恰好落在兩次檢查之間時,策略在「醒來」到「下次檢查」之間是失明的。

### 3.2 收窄手段(逐層,無單一銀彈)

因為無法讓交易所驗 token,只能多層防禦把窗口壓到可接受:

**(a) 下單前同步檢查 lease,而非只靠背景續約。** 策略在**每一次送單前**,用 monotonic clock 對本地 lease deadline 做一次內聯檢查:`now_monotonic >= lease_deadline → 拒絕下單並自殺`。這把「失明窗口」從「續約週期」壓縮到「單次下單決策的耗時」。這是最重要的一層——它讓凍結後醒來的第一筆單就被本地 deadline 擋下,不必等背景迴圈。

**(b) 偵測時鐘跳變。** 獨立看門狗比對 monotonic 與 wall-clock 的相對前進;凍結解凍後兩者會出現不一致的跳變。偵測到異常跳變(暗示剛經歷 suspend)→ 立即自殺,不信任任何既有 lease。Linux 上也可監聽 suspend/resume 事件(如 systemd-sleep hook)在 resume 時主動讓策略自殺重取 lease。

**(c) TTL 遠大於最大可容忍凍結。** T 設得足夠大,使「正常的短暫 GC / 排程延遲」不會誤觸自殺,但又要小到「M2 接管的復原時間目標(RTO)」可接受。這是張力:T 大 → 誤殺少但遷移慢;T 小 → 遷移快但正常抖動可能誤殺。交易場景建議 T 落在數十秒量級,並用 (a) 把「大 T 導致的失明」補回來——(a) 讓 T 可以設大而不放大雙開窗口。

**(d) 交易所端的冪等/去重(若有)。** 若交易所支援 client order ID 去重或單邊倉位上限,善用之——這是唯一能真正做到「下游拒絕」的槓桿,雖然不是通用 fencing。有多少用多少。

### 3.3 誠實的殘餘風險聲明

即便 (a)+(b)+(c) 全做,仍有一個理論窗口:凍結發生在「策略已通過下單前 lease 檢查(a)」與「單實際送達交易所」之間的那幾微秒~毫秒。這個窗口極小但非零,且無法靠本平台消除(需要交易所配合)。**文件必須把這句寫出來**,讓使用平台的人知道防雙開是「有界風險」而非「零風險」,並據此決定哪些策略能容忍(如小倉位、可對沖)、哪些不能(如高槓桿、單邊大倉)。

## 4. 遷移互鎖(對路徑 C)

「機器 unreachable → 是否自動遷移 S 到別台」是最危險的決策點:自動遷移若與 fencing 沒正確互鎖,就是雙開製造機。

### 4.1 硬規則:遷移必須後於 lease 確認過期

```
控制面偵測 M1 unreachable
  → 不立即遷移
  → 等待直到 (M1 最後成功續約 + T + margin_cp)     [lease 保證過期]
  → 期間 M1 若重連並續約 → 取消遷移(M1 其實活著)
  → 過期確認後 → 允許 M2 取得 S 的新 lease → M2 啟動 S
```

**絕不允許**「機器 ping 不到就馬上把策略拉到別台」這種捷徑——它跳過了 lease 過期等待,直接製造 A/B 路徑的雙開。unreachable(觀測不到)不等於 dead(確定停止);兩者之間的差距正是 lease TTL 等待要填的。

### 4.2 自動 vs 手動遷移

- **預設手動**:整機蒸發時 trader 在 UI 上把 S 指派到新機器(ARCHITECTURE §13.2)。人工介入天然帶「確認舊機真死」的判斷。
- **若要自動遷移**:必須嚴格走 §4.1 的等待,且僅對「已明確容忍 §3.3 殘餘風險」的策略開啟。自動遷移把 RTO 壓短,但把 §3 的凍結風險放到自動化路徑上,更需要 (a)+(b) 到位。
- **無論哪種**,遷移的最後一步永遠是「M2 向控制面取新 lease」,而控制面只在舊 lease 確認過期後才發——遷移不繞過 lease,遷移**就是**一次 lease 轉移。

## 5. 時鐘假設(把 §2/§3 的前提形式化)

整個 lease 安全性依賴以下假設,必須在部署環境中被保證:

| 假設 | 保證手段 | 破裂後果 |
|---|---|---|
| 控制面與 agent 的 wall-clock 偏差有界(如 < margin) | NTP,且監控 NTP 同步狀態,失步告警 | lease 過期判定錯位 → 雙開或誤殺 |
| agent 本地 monotonic clock 在非凍結時單調前進 | 內核保證(非凍結期) | — |
| 凍結可被偵測 | §3.2(b) suspend/resume hook + 跳變偵測 | 凍結後醒來的第一筆單失防 |
| lease TTL > 正常最大排程/GC 延遲 | T 選型 + §3.2(a) 補償 | 正常抖動誤殺策略 |

**margin 的具體取值**必須寫進部署配置而非硬編碼,因為它取決於實際 NTP 品質與網路抖動。建議從保守(大 margin,偏向不雙開、容忍較慢遷移)起步,依實測收緊。

## 6. 權限與爆炸半徑

安全的另一面是「當某環節被攻破,波及多大」。agent 是**持續接收遠端 DesiredState 並執行**的常駐程式,它的權限直接決定控制面被攻破 / 制品被投毒 / 串流被中間人時的爆炸半徑。

### 6.1 定調:非 root 同 user + systemd 特權啟動

- **agent 與策略進程同以一個專用非 root user 常駐**(如 `strategyagent`)。同 user 讓 pidfd 接管(RECONCILER §6/§10)最簡單——全在同一非 root user 下,不需要 `CAP_SYS_PTRACE`。
- **特權啟動交給 systemd,不手動 sudo**:unit 寫 `User=strategyagent`,systemd 以 root 執行啟動但將 agent drop 到非 root 才 exec。「以 root 啟動」與「以 root 運行」是兩回事——我們要前者(一次性、內容固定、可審計),不要後者(長期提權面)。
- 若不經 systemd 而手動啟動,agent 必須在啟動早期自行 drop privileges(root 做完一次性特權動作 → `setuid/setgid` 降權 → 之後 fork 的策略繼承非 root)。systemd `User=` 幫你正確做掉這件易錯的事,除非有硬理由,優先用它。

### 6.2 一次性特權(bootstrap / systemd)vs 常駐零特權

| 動作 | 何時 | 特權需求 |
|---|---|---|
| 放二進位、寫 systemd unit、chown `/opt/strategies` | bootstrap 一次 | root(一次性,可審計) |
| 設 cgroup delegation(`Delegate=yes`) | systemd unit | systemd 承擔 |
| cgroup 子樹內施加 per-strategy 限制 | 常駐 | **非 root**(委派子樹擁有權即可) |
| setsid + fork/exec 策略 | 常駐 | **非 root** |
| pidfd 接管(同 user) | 常駐 / 自我更新 | **非 root** |
| symlink 切換、寫 releases/ | 常駐 | **非 root**(目錄擁有權) |
| 綁 <1024 port(交易罕見) | 常駐 | 單一 `CAP_NET_BIND_SERVICE`,非整包 root |

**結論**:agent 常駐需要的不是 sudo,而是開機時授予的一組細粒度能力(目錄擁有權、delegated cgroup、按需的單一 capability)。策略進程正常**沒有任何 root 需求**(出站連交易所、讀寫自有目錄、普通 port),因此讓它們非 root 是零成本嚴格更優。

### 6.3 縱深的具體收益

agent 以非 root 常駐時,控制面被攻破 / 制品投毒的最壞後果被收斂到 `strategyagent` user 能碰的範圍:碰不到系統其餘部分、碰不到別的 user、**改不動 systemd 本身**。最後一點尤其重要——自我更新的 guard 腳本(ARCHITECTURE §11、RECONCILER §11)放在 systemd 層而非 agent 內,於是即便 agent 被完全控制,它也改不動自己的回滾邏輯。這是「guard 在 agent 之外」除了「壞 agent 起不來時仍能回滾」之外的第二個好處:壞 agent(被入侵)也繞不過回滾。

### 6.4 未決:策略間隔離

目前定調是所有策略同一個非 root user。若同機多策略且來源不完全可信,升級路徑是**每策略獨立 user**(策略間互相碰不到檔案/密鑰)。代價:pidfd 接管跨 user,需 `CAP_SYS_PTRACE` 或改接管機制,把「agent 更低特權」複雜化。這是「策略間隔離」換「接管簡單 + agent 更低特權」的取捨,屬策略密度/來源多樣性上升時的演進決策,與 per-strategy actor 同類。**單 user 是簡單起點,記為已知升級路徑。**

## 7. Secrets(與本文件相鄰的大缺口,先立邊界)

交易所 API key、簽名私鑰**不可**走「配置即版本化制品」那條路——config 制品在 S3、可版本化、可被讀。密鑰必須:

- 不落進 `releases/` 或任何制品;
- 注入方式與 config 分離(如專用 secrets 機制、開機注入環境或檔案、記憶體持有);
- 可獨立輪換,不觸發策略重新部署;
- 讀取權限受 §6.4 的 user 邊界約束(每策略獨立 user 時,密鑰隔離才真正成立)。

本文件僅立此邊界(密鑰不走制品路徑、受 user 邊界約束);完整的 secrets 生命週期設計另立文件。這是目前系統最大的功能黑洞,優先級高。

## 8. 檢查清單(上線交易前必過)

- [ ] NTP 在所有 agent 與控制面上同步,且失步會告警。
- [ ] margin_agent + margin_cp 已依實測 NTP/抖動設定,且 > 最大偏差 + 續約 RTT 抖動。
- [ ] 策略實作了**下單前同步 lease 檢查**(§3.2a),非僅背景續約。
- [ ] suspend/resume 偵測到位,resume 時策略自殺重取 lease(§3.2b)。
- [ ] lease TTL 選型經過「誤殺 vs 遷移 RTO」權衡並記錄。
- [ ] 遷移路徑(自動或手動)嚴格後於 lease 過期確認(§4.1),無 ping-fail-即遷移捷徑。
- [ ] 每個策略明確標註能否容忍 §3.3 殘餘雙開風險;不能容忍者禁用自動遷移。
- [ ] agent 以非 root user 常駐(systemd `User=`),cgroup delegation 配好。
- [ ] guard 腳本在 systemd 層,agent user 無權修改。
- [ ] 密鑰不在任何制品中,注入與 config 分離。

## 9. 明確的殘餘風險(寫給決策者)

即使以上全部到位,以下風險**無法**靠本平台單獨消除,使用者須知情並承擔:

1. **凍結後、下單前檢查與實際送單之間的微秒級窗口**(§3.3)——需交易所端 fencing 才能封閉,而交易所通常不提供。
2. **交易所端無去重時的重複單**——若交易所不支援 client order ID 去重,平台無法阻止已離開本機的單。
3. **NTP 大幅失步且未告警**——若時鐘保證破裂而監控未捕捉,lease 的所有安全論證失效。

對這三者的正確態度是:選擇性地只讓能容忍殘餘風險的策略跑在自動遷移路徑上,高風險策略保留人工確認 + 保守 TTL,並持續監控 NTP 與凍結事件。
